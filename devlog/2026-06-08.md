## 주요 결정 & 이유
- **Category는 Enum으로 고정** (BACKEND/FRONTEND/DATABASE/INFRA/CS/ETC)
  → 별도 엔티티로 분리하면 연관관계 매핑 예시는 늘지만, 한 달 일정에서는 구현 범위를 줄이는 게 우선
- **Recommendation 엔티티 추가** (추천 주제/이유/근거 요약/채택 여부 저장)
  → AI 추천 이력 기능 + 통계에 활용 가능. `based_on_summary`는 N:M 조인 대신 텍스트 스냅샷으로 저장
    (추천 시점의 컨텍스트를 그대로 보존 → 원본 기록이 수정/삭제돼도 이력이 깨지지 않음)
- **LearningRecord ↔ Tag는 `@ManyToMany` 대신 `RecordTag` 중간 엔티티로 명시적 매핑**
  → 실무에서 `@ManyToMany`를 지양하는 이유(쿼리 제어 어려움, 중간 테이블 컬럼 확장 불가)를 직접 적용해봄
- **Spring Boot 버전은 3.5.14 LTS로 고정** (Initializr 기본값은 4.0.6이었음)
  → 4.x는 아직 실무 채택률이 낮고 레퍼런스가 적어, 면접에서 "바로 투입 가능한 경험"으로 어필하기엔 3.x가 유리하다고 판단

## 적용한 JPA 디테일 (면접 어필 포인트)
- `BaseTimeEntity` + `@EnableJpaAuditing` → `@CreatedDate`/`@LastModifiedDate`로 생성·수정 시각 자동 관리
- `@Enumerated(EnumType.STRING)` → ORDINAL 대신 STRING 사용 (enum 순서 변경 시 데이터 깨짐 방지)
- 모든 연관관계 `FetchType.LAZY` + `hibernate.default_batch_fetch_size=100` → N+1 문제 인지 및 대응
- `User`–`LearningRecord`: `cascade = ALL`, `orphanRemoval = true` → 회원 삭제 시 연관 데이터 정리
- 생성자 `private` + `@Builder`, `@NoArgsConstructor(PROTECTED)` → 객체 생성 경로 강제로 일관성 확보
