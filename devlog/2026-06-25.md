## 1. S3 + CloudFront 프론트엔드 배포

### 프로덕션 빌드
- `.env.production.local`에 운영 백엔드 주소 주입 (gitignore 대상, `*.local` 패턴)
- Vite가 `npm run build` 시 `.env.production*`을 자동 로드 → 빌드 스크립트 수정 불필요

### S3 버킷
| 항목 | 값 |
|---|---|
| 버킷 이름 | `learning-hub-frontend-454252678226-ap-southeast-2-an` (전역 고유 제약 때문에 계정 ID 포함) |
| 리전 | ap-southeast-2 (기존 RDS/EC2와 통일) |
| 퍼블릭 액세스 | 차단 유지 (CloudFront OAC로만 접근 허용) |

### CloudFront 배포
- Origin access control(OAC) 선택 → "Grant CloudFront access to origin: Yes" 시 S3 버킷 정책 자동 갱신, 수동 정책 복사 불필요
- WAF는 비활성화 (월 추가 비용 발생, 포트폴리오 규모에 불필요)
- 생성된 도메인: `d3kvplpwaeztju.cloudfront.net`
- 콘솔 마법사에 Default root object 입력란이 없어서, 생성 후 General 탭에서 별도로 `index.html` 설정

### 백엔드 연동 (코드 변경 없음)
- `SecurityConfig`의 CORS `allowedOriginPatterns`에 6/18에 미리 등록해둔 `https://*.cloudfront.net` 패턴 덕분에 코드 수정 없이 즉시 통과
- EC2 `.env`의 `FRONTEND_URL`만 CloudFront 도메인으로 갱신 (OAuth 로그인 성공 후 리다이렉트 주소로 `OAuth2SuccessHandler`가 사용)

## 3. 배포 자동화

매번 빌드 → S3 콘솔 업로드 → CloudFront 무효화를 수동으로 반복하는 게 비효율적이라 스크립트로 묶음.

- `frontend/deploy.ps1`: `aws s3 sync` + `aws cloudfront create-invalidation`
- `package.json`에 `"deploy": "npm run build && powershell -ExecutionPolicy Bypass -File deploy.ps1"` 추가
- 이후로는 `npm run deploy` 한 줄로 빌드~배포~캐시 무효화 완료

### AWS CLI 설치/인증
- winget으로 AWS CLI 설치 (`winget install Amazon.AWSCLI`)
- PowerShell 새 세션에서는 PATH가 갱신 안 돼 있어 매번 `$env:Path`를 Machine+User 스코프로 재조합해야 `aws` 명령 인식됨
- `aws configure` 1차 시도 시 Access Key ID/Secret Access Key를 혼동해 잘못 입력 → `IncompleteSignature` 에러. 재발급 후 정확히 구분해서 재입력하여 해결

## 4. 트러블슈팅 ① Mixed Content

### 증상
배포 후 글쓰기 화면에서 저장 시 브라우저가 요청을 차단:
```
Mixed Content: The page at 'https://...cloudfront.net' was loaded over HTTPS,
but requested an insecure XMLHttpRequest endpoint 'http://...:8080/api/records'.
```

### 원인
프론트엔드(CloudFront)는 HTTPS인데 백엔드(EC2)는 HTTP — HTTPS 페이지에서 HTTP API를 호출하면 브라우저가 보안상 차단.

### 해결 — EC2에 nginx + Let's Encrypt 적용
1. `dnf install nginx`, `pip3 install certbot certbot-nginx` (Amazon Linux 2023엔 snap 없어서 pip 사용)
2. nginx를 리버스 프록시로 구성 (`443 → localhost:8080`)
   - 최초 시도 시 heredoc(`sudo tee ... <<'EOF'`)을 터미널에 그대로 붙여넣다가 `sudo tee` 줄 자체가 파일 내용에 섞여 들어가 `nginx -t`가 `unknown directive "sudo"` 에러. `nano`로 직접 작성해서 해결
3. `certbot --nginx -d 13.211.170.219.nip.io --non-interactive --agree-tos -m <email> --redirect`로 무료 인증서 발급 — nip.io 와일드카드 도메인이라 비용 없이 즉시 검증 통과
4. 인증서는 90일 만료라 `/etc/cron.d/certbot-renew`에 매일 새벽 3시 갱신 체크 등록
   - Amazon Linux 2023 기본 이미지엔 `cronie`가 없어서 `/etc/cron.d/` 자체가 없었음 → `dnf install cronie` 후 `systemctl enable --now crond`로 해결
5. 프론트엔드 `VITE_API_URL`/`VITE_BACKEND_URL`을 `https://...nip.io`(포트 없이 443 기본)로 변경 후 재배포

## 5. 트러블슈팅 ② OAuth redirect_uri_mismatch

### 증상
백엔드 HTTPS 적용 직후 Google 로그인이 `400: redirect_uri_mismatch`로 거부됨.

### 원인
Google Console에는 `https://...nip.io/login/oauth2/code/google`이 이미 등록되어 있었음. 그런데 Spring Boot가 nginx로부터 받은 `X-Forwarded-Proto: https` 헤더를 신뢰하지 않아, 자기 자신을 여전히 `http://...:8080`으로 인식 → `redirect_uri`를 잘못된 값으로 생성하고 있었음.

### 해결
```yaml
server:
  forward-headers-strategy: framework
```
한 줄 추가로 Spring이 리버스 프록시의 `X-Forwarded-*` 헤더를 신뢰하게 되어, Google Console에 등록된 값과 정확히 일치하는 `redirect_uri`를 생성하게 됨.

## 기타 트러블슈팅

### SSH 키 파일 권한 문제
Windows에서 받은 `.pem` 파일은 기본 권한이 너무 열려 있어 OpenSSH가 접속을 거부. `icacls`로 현재 사용자에게만 읽기 권한 부여해 해결.
```powershell
icacls learning-hub-key.pem /inheritance:r
icacls learning-hub-key.pem /grant:r "$($env:USERNAME):(R)"
```

### 운영 RDS에 로컬 데이터가 없음
로컬 PostgreSQL과 RDS는 완전히 분리된 별도 데이터베이스. `ddl-auto: update`로 스키마는 자동 생성되지만 데이터(row)는 별도 마이그레이션이 필요 — 의도된 정상 동작으로 판단, 이번엔 깨끗한 운영 환경으로 시작.

## 회고

- 인프라를 한 겹 바꾸면(HTTP→HTTPS) 그 위에 쌓인 여러 레이어(CORS, OAuth redirect, Mixed Content)가 동시에 영향을 받는다는 걸 체감했다
- 리버스 프록시 뒤의 애플리케이션은 자신이 어떤 프로토콜로 불리고 있는지 스스로 판단하지 못하므로, `X-Forwarded-*` 헤더 신뢰 설정은 프록시를 둘 때 거의 항상 같이 챙겨야 하는 항목이라는 걸 배웠다
